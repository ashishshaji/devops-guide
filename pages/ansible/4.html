Playbook Structure

A collection or list of "plays" targeting your infrastructure.

What is a "Play"?

Defines a specific set of activities for a group of hosts (e.g., a "DB Play" vs an "App Play").

  • Hosts: Target machines
  • Remote User: ubuntu, root, etc.
  • Tasks: Actions to perform
  • Become: true (Privileged)

The "Workhorses"

Modules are the actual execution units.

apt / yum service copy file

Pro Tip

"Use official documentation for module syntax; don't try to memorize it."

deploy_web.yml
--- - name: Deploy Apache Web Server hosts: web_servers become: true tasks: - name: Install Apache ansible.builtin.apt: name: apache2 state: present - name: Deploy Custom HTML ansible.builtin.copy: src: ./index.html dest: /var/www/html/index.html

1. Gathering Facts

The first step of every play. Collects system data and verifies connection before starting tasks.

2. Idempotency

Ansible checks if Apache is already installed. If yes, it does nothing (ok). If no, it installs it (changed).

Execution Command

Running the Playbook

To execute your automation logic, use the ansible-playbook CLI, ensuring you point to the correct inventory.

$ ansible-playbook -i <inventory_file> <playbook_name>.yml

The Lifecycle

Step 1: SSH Handshake

Control node connects via passwordless SSH (auth keys).

Step 2: Fact Gathering

Ansible probes the system (OS, IP, RAM) to populate variables.

Step 3: Task Execution

Tasks are executed in order. Result is changed, ok, or failed.
